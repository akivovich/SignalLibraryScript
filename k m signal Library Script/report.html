Критический #1 OnChangeFreeBlocksCount останавливает checkerProcess до его первого запуска zmvcommonlibrary.gs : 1135

Race condition: когда t_1 посылает FreeBlocksChanged к t_2, checkerProcess у t_2 ещё спит (начальный Sleep(0.5–1.0s)). Обработчик вызывает SetCheckerWorkMode(UseChecker()=false) — устанавливает m_bCheckerProcess=false. Когда поток просыпается, условие while(m_bCheckerProcess) ложно — поток выходит, не выполнив ни одной итерации. В итоге t_2 никогда не запускает processSearchNextObject(), m_bEmptyNextObject остаётся true, и вся цепочка t_2 → t_3 → t_4 разрушена.

t=0 — ResetSignal → SetCheckerWorkMode(true) → checkerProcess стартует, m_bCheckerProcess=true, поток спит 0.5–1.0s

t≈1:05 — t_1 выполняет UpdateSignalState → PostMessage(t_2, "FreeBlocksChanged") [асинхронно]

→ t_2: OnChangeFreeBlocksCount → CalcFreeBlocks: m_bEmptyNextObject=true → 0 (нет изменений)

→ t_2: SetCheckerWorkMode(UseChecker()=false) → m_bCheckerProcess=false

→ t_2: checkerProcess просыпается → while(false) → выходит!

zmvcommonlibrary.gs — OnChangeFreeBlocksCount
До
public void OnChangeFreeBlocksCount()
{
    updateVisualState(false);
    m_signal.SetCheckerWorkMode(UseChecker());
}
После
public void OnChangeFreeBlocksCount()
{
    updateVisualState(false);
    if (UseChecker()) m_signal.SetCheckerWorkMode(true);
}
✓ FreeBlocksChanged больше не останавливает checker. Когда checkerProcess просыпается, он корректно выполняет processSearchNextObject → m_bEmptyNextObject=false → CalcFreeBlocks=2 → отправляет FreeBlocksChanged к t_3 → цепочка доходит до t_4.
Критический #2 Сигнал с autoblock=false показывает R вместо Off zmvcommonlibrary.gs : 892

В processNewLensesState переход в состояние Off был обёрнут условием m_bProhodnoy. Для t_1 (IsProhodnoy()=false) при m_bAutoblockCurrent=false условие не срабатывало — код проваливался в GetNewLensesStateByFreeBlocks() → R. Дополнительная страховка на строке 919 (nNewLensesState = B) тоже не помогала, так как у t_1 уже был R.

zmvcommonlibrary.gs — processNewLensesState
До
if (m_bProhodnoy and
    !ShouldShowAutoblockLenses())
    return ZmvSignalTypes.Off;
// ... логика ...

if (!ShouldShowAutoblockLenses() and
    nNewLensesState != ZmvSignalTypes.R)
    nNewLensesState = ZmvSignalTypes.B;
После
if (!ShouldShowAutoblockLenses())
    return ZmvSignalTypes.Off;
// ... логика ...

// строки 919-920 удалены (dead code)
✓ При m_bAutoblockCurrent=false любой сигнал (не только проходной) переходит в Off. Линзы гасятся. t_2–t_4 с autoblock=true не затронуты.
Незначительный #3 GetNamedTagAsBool вместо GetNamedTagAsInt для wait-time zmvsignal.gs : 490–491

GetNamedTagAsBool возвращает bool (0 или 1). При отсутствии тега в конфиге дефолтные значения 4 и 8 трактуются как truthy → возвращается 1. В результате m_nWaitCur=1 (сек) вместо 8.

zmvsignal.gs — initWaitTime
До
m_secWaitProp =
    options.GetNamedTagAsBool("sec_wait", 4);
m_secWaitRedProp =
    options.GetNamedTagAsBool("sec_wait_red", 8);
После
m_secWaitProp =
    options.GetNamedTagAsInt("sec_wait", 4);
m_secWaitRedProp =
    options.GetNamedTagAsInt("sec_wait_red", 8);
✓ Интервал опроса checkerProcess корректен: 4 сек (обычный) и 8 сек (на красном).
Copy-paste #4 setAlsPropertiesInt — все строки присваивают m_nFr0 zmvcommonlibrary.gs : 949–955

При восстановлении свойств в редакторе (отмена изменений) все частоты АЛС записывались в m_nFr0 вместо соответствующих переменных. Не влияет на работу карты в движении, но нарушает восстановление настроек в Surveyor.

zmvcommonlibrary.gs — setAlsPropertiesInt(ZmvProperties)
До
m_nFr0  = ...GetNamedTagAsInt("fr0");
m_nFr0  = ...GetNamedTagAsInt("fr40");
m_nFr0  = ...GetNamedTagAsInt("fr60");
m_nFr0  = ...GetNamedTagAsInt("fr70");
m_nFr0  = ...GetNamedTagAsInt("fr80");
После
m_nFr0  = ...GetNamedTagAsInt("fr0");
m_nFr40 = ...GetNamedTagAsInt("fr40");
m_nFr60 = ...GetNamedTagAsInt("fr60");
m_nFr70 = ...GetNamedTagAsInt("fr70");
m_nFr80 = ...GetNamedTagAsInt("fr80");
✓ Настройки частот АЛС (fr40, fr60, fr70, fr80) корректно восстанавливаются при отмене в редакторе.
